# Webhook Event Matrix

## Overview

The Conductor receives webhook events from Gitea and translates them into orchestration actions. This document defines every supported event, the expected payload structure, and the resulting Conductor behavior.

## Webhook Configuration

### Setup

Webhooks are configured at the **organization level** in Gitea so they apply to all repositories in the `cuemarshal` org.

**Endpoint**: `https://cuemarshal.example.com/webhooks/gitea`
**Method**: POST
**Content Type**: `application/json`
**Secret**: Value of `WEBHOOK_SECRET` environment variable

### Signature Verification

Every webhook request includes an `X-Gitea-Signature` header containing the HMAC-SHA256 hex digest of the payload body, computed with the webhook secret.

```
X-Gitea-Signature: <hex-digest>
```

The Conductor accepts either raw hex or `sha256=<hex>` prefixes for compatibility. Requests with missing or invalid signatures are rejected with `401 Unauthorized`.

## Event Matrix

### Issues

#### issues.opened

A new issue has been created.

**Trigger Conditions**: Any new issue in an org repository.

**Conductor Actions**:

1. Check if the issue has a `role:*` label (manually assigned).
2. If no role label: enqueue `task:analyze` job to decompose and assign.
3. If role label present: enqueue `task:route` job to dispatch directly.
4. Create a task record in the Conductor database.
5. Push `task:created` event via WebSocket.

**Key Payload Fields**:

```json
{
  "action": "opened",
  "issue": {
    "id": 1,
    "number": 42,
    "title": "Implement user registration",
    "body": "As a user, I want to register...",
    "labels": [
      { "id": 1, "name": "role:developer", "color": "#0075ca" }
    ],
    "milestone": { "id": 1, "title": "v1.0" },
    "assignees": [],
    "state": "open"
  },
  "repository": {
    "id": 1,
    "name": "rest-api-auth",
    "full_name": "cuemarshal/rest-api-auth",
    "html_url": "https://gitea.example.com/cuemarshal/rest-api-auth"
  },
  "sender": {
    "login": "john",
    "id": 1
  }
}
```

**Filtering**: The Conductor ignores issues with the `skip-automation` label.

---

#### issues.labeled

A label was added to an issue.

**Trigger Conditions**: Label added to an open issue.

**Conductor Actions**:

1. If a `role:*` label was added: re-route the task to the new role.
2. Other labels are currently ignored by the webhook handler.

**Key Payload Fields**:

```json
{
  "action": "label",
  "issue": { "number": 42, "labels": [...] },
  "label": { "id": 5, "name": "role:architect", "color": "#e11d48" },
  "repository": { "full_name": "cuemarshal/rest-api-auth" }
}
```

---

#### issues.assigned

An issue was assigned to a user.

**Trigger Conditions**: Issue assigned to a Gitea user.

**Conductor Actions**:

- Currently ignored (not handled in the webhook router).

---

#### issues.closed

An issue was closed.

**Conductor Actions**:

- Currently ignored (task completion is handled on PR merge events).

---

### Pull Requests

#### pull_request.opened

A new pull request was created.

**Trigger Conditions**: Any new PR in an org repository.

**Conductor Actions**:

1. Check if the PR body references an issue (e.g., "Resolves #42").
2. Link the PR to the corresponding task.
3. Update task status to `review`.
4. Enqueue `review:assign` job to dispatch the reviewer agent.
5. Push `task:progress` via WebSocket with status "PR created, awaiting review".

**Key Payload Fields**:

```json
{
  "action": "opened",
  "pull_request": {
    "id": 1,
    "number": 7,
    "title": "feat(#42): Implement user registration",
    "body": "Resolves #42\n\nGenerated by developer agent using tier2.",
    "head": {
      "label": "feat/issue-42",
      "ref": "feat/issue-42"
    },
    "base": {
      "label": "main",
      "ref": "main"
    },
    "state": "open",
    "merged": false,
    "additions": 320,
    "deletions": 12,
    "changed_files": 5,
    "html_url": "https://gitea.example.com/cuemarshal/rest-api-auth/pulls/7"
  },
  "repository": {
    "full_name": "cuemarshal/rest-api-auth"
  }
}
```

---

#### pull_request_review.submitted

A review was submitted on a pull request.

**Trigger Conditions**: Review submitted (approved, changes requested, or comment).

**Conductor Actions**:

| Review Type | Action |
|-------------|--------|
| `APPROVED` | Enqueue `pr:merge` job. |
| `REQUEST_CHANGES` | Logged only (revision flow not implemented yet). |
| `COMMENT` | Ignored. |

**Key Payload Fields**:

```json
{
  "action": "submitted",
  "review": {
    "id": 1,
    "type": "pull_request_review_approved",
    "body": "LGTM! Code looks clean and well-tested.",
    "state": "APPROVED",
    "html_url": "...",
    "pull_request_url": "..."
  },
  "pull_request": {
    "number": 7,
    "title": "feat(#42): Implement user registration"
  },
  "repository": {
    "full_name": "cuemarshal/rest-api-auth"
  }
}
```

---

#### pull_request.closed

A pull request was closed (merged or unmerged).

**Conductor Actions**:

**If merged:**
1. Close the linked issue(s) via Gitea API.
2. Update task status to `completed`.
3. Delete the feature branch.
4. Check if all sub-tasks of a parent issue are done.
5. Push `pr:merged` via WebSocket.
6. Record cost metrics for the task.

**If closed without merge:**
1. Update task status to `failed`.
2. Add a comment to the linked issue explaining the closure.
3. Push `task:failed` via WebSocket.

**Key Payload Fields**:

```json
{
  "action": "closed",
  "pull_request": {
    "number": 7,
    "merged": true,
    "merged_by": {
      "login": "cuemarshal-bot"
    },
    "merge_commit_sha": "abc123..."
  },
  "repository": {
    "full_name": "cuemarshal/rest-api-auth"
  }
}
```

---

### Workflow Runs

#### workflow_run.completed

A Gitea Actions workflow run has completed.

**Conductor Actions**:

**If success:**
1. Log successful execution.
2. Update task timeline.

**If failure:**
1. Check retry count for the task.
2. If retries < max (default 3): re-dispatch the workflow with the same parameters.
3. If retries exhausted: escalate model tier (tier1→tier2, tier2→tier3).
4. If tier3 also fails: mark task as `failed`, add comment to issue, push `task:failed` via WebSocket.

**Key Payload Fields**:

```json
{
  "action": "completed",
  "workflow_run": {
    "id": 1,
    "name": "Execute Task",
    "status": "completed",
    "conclusion": "success",
    "head_branch": "feat/issue-42",
    "event": "workflow_dispatch",
    "run_started_at": "2026-02-09T14:00:00Z",
    "updated_at": "2026-02-09T14:15:00Z"
  },
  "repository": {
    "full_name": "cuemarshal/rest-api-auth"
  }
}
```

---

### Push

#### push

Code was pushed to a repository.

**Conductor Actions**:

1. If push to `main` branch: trigger test workflow if tests are configured.
2. If push to a feature branch: log in task timeline.

**Note**: Most push events are side effects of agent activity and do not require Conductor action. The Conductor checks if the push was from a bot user and skips redundant processing.

## Event Filtering

The Conductor applies these filters to avoid infinite loops and redundant processing:

### Bot User Filter

Events triggered by the `cuemarshal-bot` user (the system's own actions) are filtered:

| Event | Bot-Triggered | Action |
|-------|---------------|--------|
| `issues.opened` | Skip if part of decomposition (has `parent:*` label) | Process otherwise |
| `pull_request.opened` | Always process (triggers review) | — |
| `push` | Skip (side effect of agent work) | — |
| `pull_request_review` | Always process (triggers merge/revision) | — |

### Label-Based Filter

| Label | Effect |
|-------|--------|
| `skip-automation` | Issue is completely ignored by the Conductor |
| `manual-only` | Issue is tracked but not auto-dispatched |
| `needs-human-review` | PR is not auto-merged even if approved |
| `self-improvement` | Tracked separately in self-improvement metrics |

### Duplicate Prevention

The Conductor deduplicates events using the `X-Gitea-Delivery` header:

1. Store delivery ID in Redis with a 1-hour TTL.
2. If a delivery ID is seen again, skip processing.
3. This handles Gitea's retry mechanism for failed webhook deliveries.

## Webhook Retry Behavior

Gitea retries webhook delivery on failure:

- First retry: 10 seconds after failure
- Second retry: 60 seconds after first retry
- Third retry: 300 seconds after second retry

The Conductor must:
- Respond with `200 OK` within 30 seconds (Gitea's delivery timeout)
- Process events asynchronously (enqueue in BullMQ, respond immediately)
- Handle duplicate deliveries gracefully (idempotent processing)

## Webhook Setup Script

The `scripts/setup.sh` script creates the organization-level webhook:

```bash
# Create webhook for the organization
curl -sf -X POST \
  -H "Authorization: token ${GITEA_ADMIN_TOKEN}" \
  -H "Content-Type: application/json" \
  "${GITEA_URL}/api/v1/orgs/cuemarshal/hooks" \
  -d '{
    "type": "gitea",
    "active": true,
    "config": {
      "url": "http://conductor:4000/webhooks/gitea",
      "content_type": "json",
      "secret": "'${WEBHOOK_SECRET}'"
    },
    "events": [
      "issues",
      "issue_comment",
      "pull_request",
      "pull_request_review",
      "pull_request_comment",
      "push",
      "workflow_run"
    ]
  }'
```
